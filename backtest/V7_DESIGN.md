# Bot v7 Design Plan

백테스트가 통과한 전략만 라이브로 옮기는 게 v7의 1원칙.
v6.3d처럼 라이브 튜닝 사이클은 **종료**한다. 다음 변경은 다음 백테스트 결과로만.

## 운영 구조

### 자본 배분 (시드 빠른 증가 + 폭사 방지 균형)

```
총 시드 100%
├── A 전략 (1H 구조 스윙) ........ 60%   (앵커, 큰 손실 방어)
├── C 전략 (15m 돌파)   ........ 25%   (위성, 빈도로 복리)
└── 현금 / 펀딩 헤지 (옵션) .. 15%   (분산, 옵션 2 발동 시)
```

- A와 C는 **동시 포지션 가능** (서로 다른 알파 → 상관관계 낮음)
- A 보유 중 C 진입 가능, 단 동방향이면 합산 risk 1.5% 제한
- 한 전략이 -10% 도달 시 그 전략만 사이즈 -50% (다른 전략 정상 운영)
- 한 전략이 -20% 도달 시 그 전략 OFF (다른 전략 정상 운영)

### 레버리지 / 사이징 정책

- **risk_per_trade 기반** (사이즈%가 아님)
  - A: equity의 1% / trade
  - C: equity의 0.5% / trade
  - 손절 거리에 따라 notional 자동 조정
- **레버리지 상한** A 5x, C 3x (현재 7x보다 낮춤 — 청산 거리 확보)
- **Kelly 1/4 적용** — 백테스트 우위가 확실해도 full Kelly의 1/4만 베팅

### 자본 보호 룰 (절대 변경 금지)

1. 일일 손실 -3% 도달 → 24시간 정지
2. 주간 손실 -7% 도달 → 1주 정지
3. 월간 손실 -12% 도달 → 1달 정지 + 백테스트 재검증
4. 시드 +50% 달성 시 → 원금의 30% 인출 (다시 절대 안 잃을 안전판)
5. 시드 +100% 달성 시 → 원금의 50% 추가 인출

## 코드 구조 (v7)

```
bybit_live_bot.py            # v6.x 유지 (legacy, 비활성)
bot_v7/
├── __init__.py
├── config.py                # 환경변수, 상수
├── data_feed.py             # Bybit websocket + REST kline
├── indicators.py            # backtest/indicators.py와 공유 (DRY)
├── strategies/
│   ├── strategy_a.py        # backtest와 동일 룰 (signal-only 함수)
│   └── strategy_c.py
├── risk_manager.py          # daily/weekly/monthly stop, position sizing
├── exchange.py              # Bybit 주문 인터페이스 (idempotent)
├── state.py                 # /data 영속화 (positions, equity, kill switches)
├── notifier.py              # 텔레그램 (v6 기능 유지)
└── runner.py                # 메인 루프 (전략 ON/OFF, 헬스체크)
```

핵심 차별점 vs v6:
- 백테스트 코드와 시그널 함수 **공유** → 라이브 ↔ 백테스트 일치 보장
- 멱등 주문 (client order ID로 재시도 안전)
- 상태가 `/data/state.json`에 매분 스냅샷 (재시작 복구)
- 레버리지/사이즈 한 곳에서만 결정 (현재는 분산되어 있음)

## 단계별 마일스톤

### Phase 0 (현재)
- [x] 백테스트 인프라 + 3전략 코드 작성
- [x] 합성 데이터 자체 테스트 PASS
- [ ] **사용자가 1년치 BTC 실데이터 백테스트 실행** ← 다음 단계
- [ ] 결과 검토 + 통과 기준 점검

### Phase 1 (백테스트 결과 확보 후)
- 통과 전략만 라이브로 (A 또는 C 단독부터, 양쪽 다 통과 시 둘 다)
- v7 골격 작성 (위 코드 구조)
- 테스트넷 1주일 (실 시그널, paper money)

### Phase 2 (테스트넷 통과 후)
- 메인넷, 시드의 **30%만** 실행 (옵션 1: 안정성 검증)
- 4주간 라이브 vs 백테스트 PnL 추적 — 편차 ±20% 이내면 OK
- 월 손실 -10% 미만이면 시드의 70%로 증액

### Phase 3 (옵션 2 추가 시)
- 펀딩비 분석 임계 도달 시 캐시앤캐리 모듈 발동
- 별도 자본 풀 (15%) → A/C와 독립 운용

### Phase 4 (옵션 1+2 안정화 후)
- 멀티심볼 확장 (ETH, SOL — 단 각각 별도 백테스트 통과해야)
- Walk-forward optimization (분기별 파라미터 재검증)

## 시드 빠르게 키우기 (현실 시뮬레이션)

백테스트가 다음을 보일 때:
- A: 월 +6%, MDD 8%, Sharpe 1.5
- C: 월 +4%, MDD 5%, Sharpe 1.7

→ 통합 자본 (60% A + 25% C + 15% 현금) 기댓값:
   월 약 +5%, MDD 6%, Sharpe 1.8

→ **시드 2배까지 약 14개월, 5배까지 33개월**
   (드로다운 회복 시간 포함, fee/슬리피지 차감)

이게 빠른가? 비교군:
- BTC 단순 보유 (5년 평균 CAGR 50%): 2배 18개월
- v6.3d 현재: -35% (30일) → 음의 CAGR, 2배 영원히 도달 안 함
- "월 +30% 약속"하는 전략: 6개월 내 폭사 사례 99%

**가장 빠른 길은 폭사하지 않는 것이다.** 위 시뮬은 보수적 가정이고 실데이터에서
A의 Sharpe가 2.0+이면 월 8~12% 가능. 그 경우 **시드 2배 9개월**.

단, 라이브가 백테스트 PnL의 60%만 재현해도 성공으로 간주해야 함 (slippage,
실시간 처리 지연, 흑조 이벤트). 백테스트 +5%/월 → 라이브 +3%/월 가정이 안전선.
